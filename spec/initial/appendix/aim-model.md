# 補記: マウスジャイロ姿勢モデル

状態: 非規範  
対象: `YawPitchModel`

この文書はモデル選定の経緯を残す。実装要件と受入条件は `input.md`、`requirements.md`、`testing.md` を正本とする。

## 採用したモデル

`YawPitchModel` は、水平入力をワールド上方向まわりのyaw、垂直入力をpitchとして扱う。rollは蓄積せず、pitchを±90度より手前で制限する。

この構成はTPSカメラで一般的なyaw/pitch分離に近く、極への到達はpitch上限で防ぐ。クォータニオンを回転合成へ使う場合でも、極付近でyaw軸と視線が一致する問題はpitch上限で避ける。

## 変更内容

以前の設計案にあった次の設定は削除した。

- `virtual_radius`
- `vertical_limit`
- 固定周期を含む `raw_scale`

代わりに次を使う。

- 独立した `horizontal_sensitivity` と `vertical_sensitivity`
- `pitch_limit_degrees`
- 評価間の実経過時間

感度はユーザー向けの無次元倍率とした。角度表現を設定名へ固定せず、`count` が生のマウスセンサー値であるかのような誤解も避ける。内部では基準変換量を掛けた後、ラジアンとラジアン毎秒だけで処理する。

## 円筒モデルを採用しなかった理由

円筒モデルは、上下端へ近づくほど垂直感度を下げ、上限へ滑らかに接近できる。一方で、現在のpitchによって同じ角度移動に必要なマウス距離が変わり、上下端から中央へ戻る操作も遅くなる。

極への到達はpitch上限だけで防げるため、0.1.0では位置依存の感度曲線を導入しない。ハード上限の操作感が問題になった場合は、外向き入力だけを上限手前で減衰する `SoftPitchLimit` を独立した制約として検討する。

## 静的加速度との整合

ジャイロだけを生成すると、受信側が加速度から姿勢を推定する場合に、仮想pitchとセンサー入力が一致しない。そこで0.1.0では、同じ `YawPitchModel` のpitchから静的加速度も生成する。

Pro Controller基準の座標を +Xトリガー、+Y左、+Zボタン面外向きとすると、静止時の比力は次になる。

```text
AccelG = (-sin(pitch), 0, cos(pitch))
```

pitch=0では `(0, 0, +1) G` である。yawはワールド上方向まわりの回転なので、この値には影響しない。実物の手振りに伴う並進加速度やノイズは、照準用途に不要なため生成しない。

既存の裏取り:

- Linux `hid-nintendo` はPro Controller・左Joy-Con共通の正方向を明記し、右Joy-ConのY/Zを補正している。
- dekuNukemのSPI資料では、平置き時のZ補正値がPro Controller・左Joy-Conで正、右Joy-Conで負である。
- dekuNukemのIMU資料はaccel X/Y/Zのreport順とJoy-Con間の軸差を記録している。

右Joy-Con対応を将来追加する場合も、このモデルの座標定義は変えず、デバイス固有の軸補正を接続アダプターへ置く。

## 残る制約

Project_Demiはゲーム内カメラ角を取得しないため、仮想pitchとゲーム側のpitch上限、カメラ再中心化、照準モード切替が完全には同期しない。

0.1.0では次で対処する。

- pitch上限を設定可能にする
- 捕捉開始時と捕捉終了時に仮想pitchを0へ戻す
- プロファイル単位で実機調整できる構造を保つ
- yaw/pitchからローカルジャイロ軸への投影を単体試験で固定する
